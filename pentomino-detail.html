I. 搜尋空間複雜度極限

將 12 種五格骨牌填入 8x8 網格（扣除 4 障礙物）是經典的「精確覆蓋問題」。若使用暴力的 DFS，面對 12 種骨牌各自的 8 種變體（旋轉/翻轉），理論狀態空間高達：

$$ O(12! \times 8^{12}) $$

為了在不依賴 DLX (Dancing Links) 的前提下實現極速求解，並保留給前端動畫渲染的彈性，我設計了 優化版的回溯法 (Optimized Backtracking)，並導入嚴苛的空間幾何剪枝機制。

1. 記憶體降維：1D Array 狀態管理

傳統的演算法教學多使用二維陣列（2D Array）來表示網格，但在 JavaScript 引擎（如 V8）中，陣列中的陣列在記憶體分配上是不連續的。這會導致每一層遞迴複製陣列時，產生巨大的 GC (Garbage Collection) 負擔與尋址延遲。

我的解法是：將 8x8 網格完全扁平化為長度 64 的一維陣列。
任何 2D 座標 [row, col] 皆可透過公式 index = row * 8 + col 進行 $O(1)$ 映射，極大地提升了底層的存取效能。

2. 孤島偵測與餘數剪枝 (Flood Fill Pruning)

骨牌在擺放過程中，極容易將剩餘的空格切分成零碎的「死胡同」。如果不提早發現，演算法就會在這些註定失敗的分支上浪費數百萬次運算。為此，我導入了即時的連通區塊檢查。

pruning_logic.js
function getIslands(currentGrid) {
    const visited = new Set(); 
    
    for (let i = 0; i < 64; i++) {
        if (currentGrid[i] === null && !visited.has(i)) {
            let size = 0;
            const q = [i]; // 初始化 BFS 佇列
            visited.add(i);
            
            // 非遞迴 BFS，避免極端碎塊導致 Call Stack Overflow
            while (q.length) {
                const curr = q.shift();
                size++;
                // ... 將 4 個相鄰且合法的空格推入 Queue
            }
            
            // 🎯 核心剪枝 (Pruning Condition)
            // 孤島面積若不為 5 的倍數，此路徑絕對無解！
            if (size % 5 !== 0) return false; 
        }
    }
    return true; 
}

為何使用 BFS 而非遞迴 DFS？
在極端的碎片化盤面中，遞迴尋找連通區塊極易引發 Call Stack Overflow。透過陣列模擬佇列 (Queue) 來實作 BFS，不僅安全，還能更精準地控制記憶體。

3. Event Loop 釋放：Generator 協程運算

一般的 DFS 是一個同步的貪婪函式，它會死咬 JS Main Thread 不放導致瀏覽器卡死。為實現「解題過程視覺化」，我使用了 ES6 的 Generator 來實作協程 (Coroutine)。

async_solver.js
function* backtrack(currentGrid, pieces) {
    if (pieces.length === 0) {
        yield currentGrid; // 拋出最終勝利盤面
        return;
    }
    
    for (const variant of allVariants) {
        if (canPlace(variant, emptyIdx)) {
            placePiece(variant);
            
            if (getIslands(currentGrid)) {
                // 🎯 關鍵：暫停執行，將控制權還給主執行緒進行畫面渲染
                yield currentGrid; 
                yield* backtrack(currentGrid, remainingPieces);
            }
            removePiece(variant); // Backtracking
        }
    }
}

II. 前端渲染架構與互動防護

將底層抽象的數學陣列轉化為玩家可絲滑拖放的 DOM 元素，考驗的是前端工程師對 瀏覽器渲染機制 (Rendering Pipeline) 的掌握，以及對人機介面 (HCI) 體驗的極致追求。

1. 真・物理重心自轉演算法 (Center of Mass Rotation)

在實作「滑鼠滾輪旋轉方塊」時，我遇到了經典的 Pivot Illusion (錨點錯覺) 挑戰。如果不對稱的骨牌（如 P 或 U 形）繞著外框幾何中心旋轉，會產生極大的迴轉半徑，且長寬互換的 0.5 格偏移會導致方塊「越轉越往右下角飄走」。

center_of_mass_rotation.js
// 1. 計算當前陣列的「物理重心」(Center of Mass)
// 每一塊骨牌固定 5 格，重心座標 = [總和R / 5, 總和C / 5]
const getCoM = (blocks) => {
    let sr = 0, sc = 0;
    blocks.forEach(([r, c]) => { sr += r; sc += c; });
    return [sr / 5, sc / 5];
};

const [oldComR, oldComC] = getCoM(oldBlocks);
const [newComR, newComC] = getCoM(newBlocks);

// 🌟 數學奇蹟：因為 N=5 是奇數，除法尾數只會是 .0, .2, .4, .6, .8
// Math.round 絕對不會遇到 .5 的模糊地帶，實現 100% 零誤差的對稱進位！
const targetGridR = oldGridR + Math.round(oldComR - newComR);
const targetGridC = oldGridC + Math.round(oldComC - newComC);

這段代碼的數學意義在於：它讓骨牌拋棄了虛擬的外框，改為「以真實的質量中心作為自轉軸」，完美解決了旋轉時的視覺位移 Bug。

2. 貪婪圖形著色演算法 (Greedy Graph Coloring)

為了確保通關後生成的戰績卡與 Emoji 複製文字中，「相鄰的骨牌絕對不會是同一種顏色」，我將 8x8 網格抽象為 無向平面圖 並應用了圖論演算法：

greedy_graph_coloring.js
// 建立相鄰的邊 (Edge) 形成 Graph
Object.keys(pieces).forEach(p => {
    const usedColors = new Set();
    // 收集所有鄰居已經使用的顏色
    adj[p].forEach(neighbor => {
        if (colorMap[neighbor]) usedColors.add(colorMap[neighbor]);
    });

    // 篩選出合法的顏色池
    const available = ALL_COLORS.filter(c => !usedColors.has(c));

    // 🎯 貪婪選擇：從合法池中，優先挑選「全域使用次數最少」的顏色
    available.sort((a, b) => colorUsage[a] - colorUsage[b]);
    
    const chosen = available[0];
    colorMap[p] = chosen;
    colorUsage[chosen]++;
});

這個演算法不僅消除了視覺上的相鄰同色黏著感，更強制讓 7 種顏色均勻分布在 12 塊骨牌上，展現出極致的資料視覺化美感。

3. 動態佈局與狀態同步 (Responsive State Sync)

響應式網頁最大的痛點是：當使用者拉扯視窗時，絕對定位的方塊會跑版。我利用了 ResizeObserver 建立狀態同步機制：不依賴 CSS 的百分比，而是將方塊鎖定在底層的 1D Array 中。當視窗改變，立即根據網格的全新 BoundingClientRect 與陣列內的 [row, col]，重新推算出絕對的 Pixels，達到零死角的響應式防護。
