Pentomino: Algorithm Optimization & Frontend Mechanics

I. 搜尋空間複雜度極限

將 12 種五格骨牌填入 8x8 網格（扣除 4 障礙物）是經典的「精確覆蓋問題」。若使用暴力的 DFS，面對 12 種骨牌各自的 8 種變體（旋轉/翻轉），理論狀態空間高達：

$$ O(12! \times 8^{12}) $$

為了在不依賴 DLX (Dancing Links) 的前提下實現極速求解，並保留給前端動畫渲染的彈性，我設計了 優化版的回溯法 (Optimized Backtracking)，並導入嚴苛的空間幾何剪枝機制。

1. 記憶體降維：1D Array 狀態管理

傳統的演算法教學多使用二維陣列（2D Array）來表示網格，但在 JavaScript 引擎（如 V8）中，陣列中的陣列在記憶體分配上是不連續的。這會導致每一層遞迴複製陣列時，產生巨大的 GC (Garbage Collection) 負擔與尋址延遲。

我的解法是：將 8x8 網格完全扁平化為長度 64 的一維陣列。
任何 2D 座標 [row, col] 皆可透過公式 index = row * 8 + col 進行 $O(1)$ 映射，極大地提升了底層的存取效能。

2. 孤島偵測與餘數剪枝 (Flood Fill Pruning)

骨牌在擺放過程中，極容易將剩餘的空格切分成零碎的「死胡同」。如果不提早發現，演算法就會在這些註定失敗的分支上浪費數百萬次運算。為此，我導入了即時的連通區塊檢查。

pruning_logic.js
function getIslands(currentGrid) {
    const visited = new Set(); // 紀錄已走訪的格子
    
    for (let i = 0; i < 64; i++) {
        // 尋找尚未被填補，且還沒走訪過的空格作為起點
        if (currentGrid[i] === null && !visited.has(i)) {
            let size = 0;
            const q = [i]; // 初始化 BFS 佇列
            visited.add(i);
            
            // 執行非遞迴廣度優先搜尋 (BFS)，探索整個連續區塊
            while (q.length) {
                const curr = q.shift();
                size++;
                // 略：檢查上下左右 4 個相鄰格，若合法且為空則推入 Queue
            }
            
            // 🎯 核心剪枝邏輯 (Pruning Condition)
            // 每塊骨牌面積皆為 5。若孤島面積不是 5 的倍數，此盤面絕對無解！
            if (size % 5 !== 0) return false; 
        }
    }
    return true; // 全部的連通區塊都合法，允許繼續遞迴
}

為何使用 BFS 而非遞迴 DFS？
因為在極端的碎片化盤面中，遞迴尋找連通區塊極易引發 Call Stack Overflow。透過陣列模擬佇列 (Queue) 來實作 BFS，不僅安全，還能更精準地控制記憶體。

3. Event Loop 釋放：Generator 協程運算

一般的 DFS 是一個同步且貪婪的函式。一旦執行，它會死咬著 JS 的 Main Thread 不放，導致瀏覽器畫面完全凍結，使用者會以為網頁當機了。為了解決這個問題並實現「解題過程視覺化」，我使用了 ES6 的 Generator。

async_solver.js
// 使用 function* 定義生成器函式，將其轉變為可暫停的協程 (Coroutine)
function* backtrack(currentGrid, pieces) {
    // Base Case: 所有骨牌都放置完畢
    if (pieces.length === 0) {
        yield currentGrid; // 拋出最終勝利盤面
        return;
    }
    
    const emptyIdx = currentGrid.indexOf(null);
    
    for (const variant of allVariants) {
        if (canPlace(variant, emptyIdx)) {
            placePiece(variant); // 試著放置骨牌
            
            // 結合孤島偵測進行先期診斷
            if (getIslands(currentGrid)) {
                
                // 🎯 關鍵：暫停執行，將控制權還給主執行緒！
                yield currentGrid; 
                
                // 遞迴進入下一層搜尋，並傳遞 yield
                yield* backtrack(currentGrid, remainingPieces);
            }
            removePiece(variant); // 復原狀態 (Backtracking)
        }
    }
}

這段程式碼的精妙之處：
遇到 yield 時，函式會將當前陣列狀態拋出並「暫停」。這時，外層負責 UI 渲染的邏輯可以接手，利用 requestAnimationFrame 或 setTimeout(..., 0) 繪製出當前的畫面，接著再呼叫 iterator.next() 喚醒演算法繼續跑。這在不使用 Web Worker 的情況下，完美實現了非阻塞的高效能運算動畫。

II. 前端渲染架構與互動防護

將底層抽象的數學陣列轉化為玩家可絲滑拖放的 DOM 元素，考驗的是前端工程師對 瀏覽器渲染機制 (Rendering Pipeline) 的掌握，以及對跨裝置原生事件的防護能力。

1. GPU 硬體加速與 Layout Thrashing 防護

在畫面上同時拖曳並更新多達 12 塊結構複雜的 DOM 元素，如果我們使用修改 style.top 與 style.left 的方式來實現，會不斷觸發瀏覽器的 Reflow (重排)，導致嚴重的掉幀卡頓 (Jank)。

我的解法是：強制圖層提升 (Composite Layer Promotion)。

gpu_acceleration.css
/* 拖曳狀態的專屬類別：強制觸發 Hardware Acceleration */
.piece-container.dragging {
    z-index: 9999 !important;
    cursor: grabbing;
    
    /* 關閉所有過渡效果，確保骨牌能 0 延遲跟隨手指/滑鼠 */
    transition: none; 
    
    /* 提前提示瀏覽器預留 GPU 記憶體空間 */
    will-change: transform, left, top;
    
    /* 🎯 核心：使用 translateZ(0) 欺騙瀏覽器開啟 3D 渲染，
       將該元素剝離主圖層，交由 GPU 直接進行 Composite 疊加 */
    transform: scale(1.05) translateZ(0) !important;
}

2. 行動端原生手勢防護 (Mobile Safari Defenses)

Web Game 在手機上的最大敵人就是瀏覽器的原生手勢。例如 iOS Safari 著名的「雙擊放大 (Double Tap to Zoom)」功能，會讓玩家在快速旋轉或放置骨牌時，畫面突然失控放大。為此，我實作了時間差攔截器。

mobile_safari_fix.js
let lastTouchEnd = 0;

// 終極防護：掛載全域事件監聽器
// 設置 passive: false 以允許使用 preventDefault()
document.addEventListener('touchend', (event) => {
    const now = (new Date()).getTime();
    
    // 如果兩次點擊事件的間隔時間小於 300 毫秒 (系統預設的雙擊判定閾值)
    if (now - lastTouchEnd <= 300) {
        // 判定為誤觸雙擊，強制阻斷事件冒泡與原生行為
        event.preventDefault();
    }
    
    lastTouchEnd = now;
}, { passive: false });

3. 圖形著色演算法 (Greedy Graph Coloring)

當玩家通關時，系統會自動生成一張精美的戰績圖與文字版成績單（Emoji）。為了確保畫面上「相鄰的骨牌絕對不會是同一種顏色」，我實作了經典的圖論演算法，保證極致的視覺體驗。

  • 建立鄰接表 (Adjacency List): 掃描完成的 8x8 網格。如果某格是骨牌 A，且它的右邊或下邊是骨牌 B，就在兩者間建立一條 Edge，形成一個無向平面圖。
  • 貪婪著色 (Greedy Approach): 遍歷所有節點，取得其鄰居「已經使用過」的顏色集合。接著從預設的 7 種主題色中，選出一個 「全域使用次數最少」 且 「不在鄰居使用清單中」 的顏色。這不僅解決了視覺黏著，更強制確保了顏色平均分配。
