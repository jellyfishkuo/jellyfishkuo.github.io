Pentomino: Algorithm Optimization & Frontend Mechanics

I. 搜尋空間複雜度極限

將 12 種五格骨牌填入 8x8 網格（扣除 4 障礙物）是經典的「精確覆蓋問題」。若使用暴力的 DFS，面對 12 種骨牌各自的 8 種變體（旋轉/翻轉），理論狀態空間高達：

$$ O(12! \times 8^{12}) $$

為了在不依賴 DLX (Dancing Links) 的前提下實現極速求解，並保留給前端動畫渲染的彈性，我設計了 優化版的回溯法 (Optimized Backtracking)，並導入嚴苛的空間幾何剪枝機制。

1. 記憶體降維：1D Array 狀態管理

傳統的演算法教學多使用二維陣列（2D Array）來表示網格，但在 JavaScript 引擎（如 V8）中，陣列中的陣列在記憶體分配上是不連續的。這會導致每一層遞迴複製陣列時，產生巨大的 GC (Garbage Collection) 負擔與尋址延遲。

我的解法是：將 8x8 網格完全扁平化為長度 64 的一維陣列。
任何 2D 座標 [row, col] 皆可透過公式 index = row * 8 + col 進行 $O(1)$ 映射，極大地提升了底層的存取效能。

2. 孤島偵測與餘數剪枝 (Flood Fill Pruning)

骨牌在擺放過程中，極容易將剩餘的空格切分成零碎的「死胡同」。如果不提早發現，演算法就會在這些註定失敗的分支上浪費數百萬次運算。為此，我導入了即時的連通區塊檢查。

pruning_logic.js
function getIslands(currentGrid) {
    const visited = new Set(); 
    
    for (let i = 0; i < 64; i++) {
        if (currentGrid[i] === null && !visited.has(i)) {
            let size = 0;
            const q = [i]; // 初始化 BFS 佇列
            visited.add(i);
            
            // 非遞迴 BFS，避免極端碎塊導致 Call Stack Overflow
            while (q.length) {
                const curr = q.shift();
                size++;
                // ... 將 4 個相鄰且合法的空格推入 Queue
            }
            
            // 🎯 核心剪枝 (Pruning Condition)
            // 孤島面積若不為 5 的倍數，此路徑絕對無解！
            if (size % 5 !== 0) return false; 
        }
    }
    return true; 
}

為何使用 BFS 而非遞迴 DFS？
在極端的碎片化盤面中，遞迴尋找連通區塊極易引發 Call Stack Overflow。透過陣列模擬佇列 (Queue) 來實作 BFS，不僅安全，還能更精準地控制記憶體。

3. Event Loop 釋放：Generator 協程運算

一般的 DFS 是一個同步的貪婪函式，它會死咬 JS Main Thread 不放導致瀏覽器卡死。為實現「解題過程視覺化」，我使用了 ES6 的 Generator 來實作協程 (Coroutine)。

async_solver.js
function* backtrack(currentGrid, pieces) {
    if (pieces.length === 0) {
        yield currentGrid; // 拋出最終勝利盤面
        return;
    }
    
    for (const variant of allVariants) {
        if (canPlace(variant, emptyIdx)) {
            placePiece(variant);
            
            if (getIslands(currentGrid)) {
                // 🎯 關鍵：暫停執行，將控制權還給主執行緒進行畫面渲染
                yield currentGrid; 
                yield* backtrack(currentGrid, remainingPieces);
            }
            removePiece(variant); // Backtracking
        }
    }
}

II. 前端渲染架構與互動防護

將底層抽象的數學陣列轉化為玩家可絲滑拖放的 DOM 元素，考驗的是前端工程師對 瀏覽器渲染機制 (Rendering Pipeline) 的掌握，以及對人機介面 (HCI) 體驗的追求。

1. 人體工學互動：滾輪截流旋轉 (Scroll Throttling)

為提升電腦版使用者的操作流暢度，我實作了「滑鼠滾輪控制骨牌旋轉」的直覺操作。但如果將 rotate() 直接綁定在 wheel 事件上，高精度的滑鼠或 Mac 觸控板會在一秒內發射數十次事件，導致方塊像直升機一樣狂轉。

我的解法是：實作精準的截流 (Throttle) 機制，並阻斷預設的畫面滾動。

ergonomic_scroll.js
document.addEventListener('wheel', (e) => {
    const targetPiece = activeDragPiece || selectedPiece;
    if (!targetPiece || e.deltaY === 0) return;

    // 阻斷網頁因滾輪而上下滑動，鎖定玩家視線
    e.preventDefault(); 

    // 🎯 Throttle (節流) 邏輯：鎖定 150ms 的冷卻時間
    if (targetPiece._isRotating) return;
    targetPiece._isRotating = true;
    
    setTimeout(() => { 
        targetPiece._isRotating = false; 
    }, 150);

    targetPiece.rotate(); // 安全觸發矩陣變換與 UI 重繪
}, { passive: false });

2. 動態佈局與狀態同步 (Responsive State Sync)

響應式網頁面臨最大的挑戰是：「當使用者在遊玩途中調整視窗大小，放置好的方塊座標該如何更新？」
絕對不能讓方塊跑出版面。因此，我使用了 ResizeObserver 建立一套 **State-to-View** 的絕對同步機制：

  • 不依賴 CSS 的百分比定位，而是將方塊的真實位置儲存在底層的 1D Array 中。
  • 當 ResizeObserver 監聽到網格尺寸改變，立即重新抓取 board.getBoundingClientRect()，根據新的網格寬度與陣列中的 [row, col]，重新推算出每一個方塊該出現的精確 Pixels。

3. GPU 硬體加速與 Layout Thrashing 防護

拖曳複雜的 DOM 元素時，修改 top/left 會引發嚴重的重新排版 (Reflow) 造成效能瓶頸（Jank）。

gpu_acceleration.css
/* 拖曳狀態專屬類別：強制觸發 Composite Layer 提升 */
.piece-container.dragging {
    z-index: 9999 !important;
    cursor: grabbing;
    transition: none; /* 關閉過渡動畫防止拖影 */
    will-change: transform; /* 提示預留 GPU 記憶體 */
    
    /* 使用 translateZ(0) 將元素交由 GPU 獨立渲染 */
    transform: scale(1.05) translateZ(0) !important;
}

4. 圖形著色演算法 (Greedy Graph Coloring)

通關生成戰績卡時，為確保「相鄰骨牌絕對不同色」，我將 8x8 網格抽象為 **無向平面圖 (Planar Graph)**，實作圖論中的貪婪著色 (Greedy Coloring)。從 7 種主題色中，選出「全域使用最少」且「鄰居未使用」的顏色，解決相鄰同色的視覺混淆，並讓戰績 emoji 分布極具美感。
